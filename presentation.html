The Challenge

Our expertise is in meetings.
It should be in the firm.

01
Decisions made in meetings — lost in email threads Agreed positions exist only in attendees' heads and inboxes. No durable record.
02
QSAs giving inconsistent guidance on the same requirement Entity A gets one answer. Entity B gets a different one. Both cite "Patronusec policy."
03
No audit trail when a decision is challenged in a QA review Who agreed this interpretation? When? Based on what? Currently — no record.

ROC wording templates recreated per-assessor, not standardised Every QSA writes the Section 6.2 sampling narrative from scratch. Inconsistency in every delivery.

Architecture

Two article types. One article is not enough.

Interpretation Article
"What does this requirement mean in this scenario?"
  • Full STAR narrative — Situation, Task, Action, Result
  • States the firm's agreed position unambiguously
  • Includes conditions on when it applies
  • Documents assessment impact — what the QSA must verify
  • Traceable to a named meeting and approver
id: "DSS-8.3.6-001"
article_type: "interpretation"
status: "agreed"
meeting_id: "2026-02-12-monthly"
// Result: "The 8-char fallback covers length
// only. Complexity gap requires 3 controls..."
Wording Guidance Article
"What should we write in the ROC/SAQ?"
  • Multiple named variants — Generic, Shorter, Cloud-native…
  • % Placeholder_Name % convention for entity-specific values
  • Covers ROC table cell wording as well as free-text narratives
  • "What not to write" section — captures rejected wording
  • Always paired with a companion interpretation article
id: "DSS-11.3.1-001"
article_type: "wording-guidance"
status: "agreed"
// Variants: Generic · Shorter · Cloud-native
// Container/CI-CD · ROC Table S5.3

The original brief conflated these two types. Keeping them separate means QSAs can quickly find the approved text they need without re-reading the full analysis — while the interpretation article remains the authoritative record for QA audit purposes.

P2PE uses hyphen-notation for requirements (1-4, 6-4) rather than dot-notation. The schema handles this cleanly — article IDs follow the standard's own numbering convention.

This article is a draft. It was built directly from your existing submission and contains the proposed agreed position. This meeting could agree it today.

# The question:
# Section 4 says "every N/A requirement
# must be in this table" — but does that
# mean only top-level, or sub-reqs too?

Situation: 1-4.b marked N/A (PTS device,
not FIPS HSM). Parent 1-4 is In Place.

Proposed position:
The instruction applies to ALL requirements
and sub-requirements marked N/A —
not only top-level requirements.

Mixed cases (e.g. 6-4.b):
→ Partial entry required in Section 4
→ Identify the specific N/A element
→ Include justification for that element

Why this matters for QA: Without a firm position, QSAs are currently inconsistent on this across P-ROV deliverables — some list sub-requirement N/As in Section 4, some do not. A QA reviewer or PCI SSC audit could flag the inconsistency.

Technical Approach

Three options. One recommended path.

MVP · Start Here
Option A
SharePoint Native
Articles as SharePoint Pages with metadata columns. Term Store for taxonomy. Microsoft Search for discovery.
Zero additional infrastructure — already in M365
Microsoft Search works immediately
Familiar to all QSAs
Page versioning = automatic audit trail
WYSIWYG editing invites format drift
No clean diff view between versions
Long-Term · Best Governance
Option B
Git → SharePoint
Articles authored in Markdown in this repo. GitHub Actions publishes to SharePoint via Graph API on merge. PR = approval gate.
Git history is tamper-evident — ideal for QA audit
PR-based approval is rigorous and named
Schema linter validates fields on every commit
This repo is already set up for this approach
QSAs need Git / GitHub web editor familiarity
Pipeline requires maintenance
Option C
Dedicated Static Site
Docusaurus or VitePress builds a documentation portal. Hosted on Azure Static Web Apps or GitHub Pages.
Best browsing experience; Algolia search
Built-in versioning by standard version
Outside M365 — separate access control
Most engineering effort to maintain
May face M365-only policy resistance

Recommendation: Start with Option A — prove value within weeks, zero cost, familiar for QSAs. Migrate to Option B in Phase 2 using this repo as the source of truth. The groundwork is already in place.

Gap Analysis

Ten things the original brief
didn't cover.

1
No approval workflow Without a named approver, a KB article is opinion — not a firm position.
2
No article lifecycle PCI standards change. Stale articles with no review_due are dangerous.
3
No attribution meeting_id + approved_by make decisions defensible in a QA audit.
4
Two article types conflated Interpretation decisions and ROC wording templates serve different audiences.
5
No supersession chain Old articles must stay visible with a banner — never deleted. Audit trail requirement.
6
PCI SSC IP & copyright contains_standard_excerpts flag triggers IP review. Disclaimer required on every article.
7
No cross-references Requirements interact. Related articles must link to each other.
8
No change notifications QSAs on active assessments need to know when guidance changes.
9
No assessment tool integration The KB is most valuable from within the assessment workflow, not a separate URL.
10
No controlled tag taxonomy Free-text tags degrade. Term Store or enum list must be established from day one.
